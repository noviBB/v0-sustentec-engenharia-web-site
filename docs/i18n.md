# i18n

The single most important contract in this codebase: **no user-facing string is hardcoded in JSX**. Everything goes through `t("key")`.

## How it works

[lib/language-context.tsx](../lib/language-context.tsx) defines:

- A `Language` type: `"pt" | "en"`.
- A `translations` object: `{ pt: { [key]: string }, en: { [key]: string } }`. ~80 keys at last count, namespaced informally (`nav.*`, `hero.*`, `stats.*`, `services.*`, `tracker.*`, `values.*`, `contact.*`, `footer.*`).
- A `LanguageProvider` (mounted in [app/layout.tsx](../app/layout.tsx)) that holds the current language in `useState`, defaulting to `"pt"`.
- A `useLanguage()` hook returning `{ language, setLanguage, t }`.

The `t(key)` function looks up `translations[language][key]`. **On miss, it returns the key itself** — there is no fallback chain to PT, no warning, no error. A typo will silently render the literal key string to the page.

## PT-first policy

Portuguese is the source of truth. Author every new string in PT first, then translate to EN. The site's primary audience is in Rio de Janeiro state; EN exists for international visitors.

## Adding a key — checklist

1. Pick a namespace + name: `section.thing` (e.g. `contact.success`).
2. Add to the `pt` dictionary first with the authored Portuguese.
3. Add the **same key** to the `en` dictionary with the translation.
4. Use it at the call site: `t("section.thing")`.
5. Search for any hardcoded version of the same text in JSX and replace.

Both dictionaries must be updated **atomically in the same commit**. A key in `pt` but not `en` will render the literal `"section.thing"` to English visitors.

## Switcher

The language switcher lives in [components/header.tsx](../components/header.tsx) and calls `setLanguage("pt" | "en")`. State is **in-memory only** — refreshing the page resets to PT. Persisting the choice (cookie or `localStorage`) and updating `<html lang>` are tracked in [roadmap.md](roadmap.md).

## Future direction

The dictionaries are inline TypeScript objects today. As the surface grows, consider extracting `pt` and `en` to JSON files (`lib/i18n/pt.json`, `lib/i18n/en.json`) for better editor support and easier diffs. Not urgent at current scale.
